# Tcloud

Tcloud — небольшой файловый «облачный» сервис на PHP. Проект состоит из HTTP API для сервера и CLI-клиента для пользователя. Идея простая: пользователь работает с файлами из терминала, а фактическое хранение больших данных вынесено в Telegram через Bot API. На стороне сервера остаются метаданные, структура каталогов и идентификаторы чанков.

В проекте есть важные практические вещи: авторизация по bearer-токену, разбиение файлов на части, проверка SHA-256, клиентское шифрование, докачка/дозагрузка через локальный трекинг прогресса.

## Зачем нужен проект

Tcloud решает задачу удалённого хранения файлов с минимальной серверной частью:

- пользователь загружает файл или каталог через CLI;
- клиент упаковывает данные в `tar.gz`, шифрует их AES-256-GCM и режет на чанки;
- сервер принимает чанки, проверяет хеш и отправляет каждый чанк в Telegram как документ;
- в MySQL сохраняются метаданные файла и Telegram `file_id` каждого чанка;
- при скачивании клиент получает список чанков, скачивает их обратно, расшифровывает и распаковывает файл.

В результате сервер не хранит исходные пользовательские данные в открытом виде. Он хранит только служебные файлы, метаданные и временный кеш скачанных чанков.

## Что умеет

CLI-клиент поддерживает команды:

- `ls` — показать содержимое удалённого каталога;
- `mkdir` — создать удалённый каталог;
- `upload` — загрузить локальный файл или каталог в Tcloud;
- `download` — скачать удалённый файл обратно на компьютер;
- `rm` — удалить файл или каталог на сервере;
- `config` — настроить адрес API, токен и ключ шифрования.

Серверная часть предоставляет отдельные PHP-эндпоинты для создания файлов, загрузки чанков, получения метаданных, скачивания чанков, просмотра каталогов и удаления.

## Архитектура в двух словах

```text
Пользователь
   │
   ▼
CLI-клиент: cli_client/main.php
   │  tar.gz + AES-256-GCM + chunks
   ▼
HTTP API: api/*.php
   │  bearer auth + MySQL metadata
   ▼
Telegram Bot API
```

### Поток загрузки

1. Пользователь вызывает `tcloud upload <local_path> <remote_path>`.
2. Клиент архивирует локальный файл или каталог в `tar.gz`.
3. Архив режется на части до 20 МБ с учётом служебных байтов шифрования.
4. Каждый чанк шифруется на клиенте AES-256-GCM.
5. Клиент отправляет чанк на `/add_chunk.php` вместе с SHA-256-хешем.
6. Сервер проверяет хеш, отправляет чанк в Telegram и сохраняет метаданные в MySQL.

### Поток скачивания

1. Пользователь вызывает `tcloud download <remote_path> <local_path>`.
2. Клиент запрашивает `/meta.php`, чтобы получить id файла и список чанков.
3. Для каждого чанка клиент вызывает `/chunk.php`.
4. Сервер при необходимости скачивает чанк из Telegram во временный каталог `api/downloads`.
5. Клиент забирает временный файл, расшифровывает данные и собирает архив.
6. После распаковки архив превращается обратно в исходный файл или каталог.

## Структура проекта

```text
.
├── api/
│   ├── add_chunk.php          # Приём зашифрованного чанка и отправка в Telegram
│   ├── chunk.php              # Получение чанка из Telegram и выдача временного пути
│   ├── downloads/
│   │   └── cleanup.php        # Очистка временных скачанных чанков
│   ├── functions.php          # Общие функции API: auth, DB, Telegram, пути, JSON-ответы
│   ├── ls.php                 # Список файлов и каталогов в удалённом хранилище
│   ├── meta.php               # Метаданные файла и список его чанков
│   ├── mkfile.php             # Создание удалённого каталога или файла-заглушки
│   └── remove.php             # Удаление файлов и каталогов
├── cli_client/
│   ├── functions.php          # Логика CLI: upload/download/config/crypto/API calls
│   └── main.php               # Точка входа CLI-клиента и разбор команд
└── README.md
```

## Серверная часть

API написано на PHP и рассчитано на размещение внутри `public_html`. Это важно: код ищет внешний каталог проекта, поднимаясь по дереву до папки `public_html`, а затем ожидает рядом с ней служебные директории `secret` и `storage`.

Ожидаемая структура на сервере примерно такая:

```text
project-root/
├── public_html/
│   └── api/
├── secret/
│   ├── bearer.token
│   ├── dbuser.pswd
│   ├── tgbot.id
│   └── tgchat.id
└── storage/
```

### Секреты

Файлы в `secret/` используются так:

- `bearer.token` — токен для авторизации клиентов в API;
- `dbuser.pswd` — пароль MySQL-пользователя `tcloud`;
- `tgbot.id` — токен Telegram-бота в формате, который можно добавить после слова `bot` в URL Telegram API;
- `tgchat.id` — id чата, куда бот отправляет документы-чанки.

## CLI-клиент

Точка входа клиента — `cli_client/main.php`. Его удобно запускать напрямую через PHP или обернуть в shell-скрипт/симлинк `tcloud`.

Клиент хранит локальную конфигурацию в `~/.tcloud`:

- `address` — базовый URL API;
- `token` — bearer-токен;
- `key` — 32-байтовый ключ шифрования;
- `tracking/` — временные данные для продолжения прерванных upload/download-операций.

Важно: ключ шифрования генерируется и хранится только на клиенте. Если потерять файл `~/.tcloud/key`, ранее загруженные данные нельзя будет расшифровать.

## Зависимости

Нужны:

- PHP с расширениями `curl`, `openssl`, `pdo_mysql`, `pcntl`;
- MySQL/MariaDB;
- утилита `/usr/bin/tar`;
- Telegram-бот и чат, куда бот может отправлять документы;
- веб-сервер, умеющий выполнять PHP.

## Безопасность и ограничения

- Авторизация API сделана через `Authorization: Bearer <token>`.
- Данные шифруются на клиенте перед отправкой на сервер.
- Для каждого чанка считается SHA-256, сервер проверяет целостность перед отправкой в Telegram.
- Размер чанка ограничен примерно 20 МБ, потому что используется отправка документов через Telegram Bot API.
- Сервер намеренно ограничивает частоту обращений к Telegram через файл `api/downloads/usetime`, чтобы не делать запросы чаще одного раза в секунду.
- Имена новых файлов ограничены латиницей, цифрами, точкой, подчёркиванием и дефисом.
